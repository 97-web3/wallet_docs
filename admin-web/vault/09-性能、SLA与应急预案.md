# Vault / 性能、SLA 与应急预案

> 返回 [Vault 资金管理索引](../10-Vault资金管理.md)

---

## 1. 规模与性能目标（建议写入产品 SLA）

- **规模**：50+ Vault（或按网络 10+，资产 100+）
- **刷新延迟**：余额/同步状态 < 30 秒（异常时可降级并标注“数据陈旧”）
- **并发**：多管理员同时查看与配置不冲突（配置需写入审计，并避免覆盖写）
- **历史**：调整记录与交易/快照支持按月/季度检索与导出
- **告警**：高频触发时需去重/冷却，避免告警风暴

---

## 2. 前端性能建议（文档级）

- 列表虚拟化/分页：避免一次渲染全部网络/资产
- 局部刷新：按网络卡片独立刷新，失败不阻塞全局
- 数据新鲜度：统一显示“最后更新时间/blocks_behind”

---

## 3. 应急预案（Runbook，建议最小集合）

### 3.1 余额 critical（资金不足）

1. 确认数据新鲜度（`last_sync_at`/`blocks_behind`）
2. 查看 pending 流出（待处理提现/链上 pending）
3. 决策：
   - 冷钱包补充 / 交易所转入 / 临时限流出款
4. 记录：创建事件工单，关联告警与操作人

### 3.2 同步失败/延迟（数据不可用）

1. 切换到“系统状态监控”确认节点/索引器状态（见 `wallet_docs/admin-web/14-系统状态监控.md`）
2. 触发手动同步重试
3. 进入降级模式：标注“数据陈旧”，暂停依赖余额的自动决策（如有）

### 3.3 误配置阈值/通知

1. 回滚到上一个已知正确版本（需要“配置变更审计”支持）
2. 通知相关责任人，复盘原因与影响范围
